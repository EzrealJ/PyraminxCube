# RBAC 术语表

> 本文档统一解释 RBAC 权限管理框架中的专业术语，便于快速查阅。
> 与 [RBAC_Design.md](./RBAC_Design.md) 配合使用。

---

## 核心概念

### RBAC

**全称**：Role-Based Access Control（基于角色的访问控制）

**解释**：一种权限管理模型，通过角色作为用户和权限之间的桥梁，简化权限管理。

```
用户 → 角色 → 权限
```

---

### 功能权限

**解释**：控制用户能"做什么"（操作能力），属于技术层面的控制。

| 类型 | 说明 |
|------|------|
| API 权限 | 后端接口访问权限 |
| 功能按钮权限 | 前端页面按钮/元素的显示控制 |

**框架实现**：完整实现 ✅

---

### 数据权限

**解释**：控制用户能"看到什么"（数据范围），属于业务层面的控制。

**框架实现**：仅提供抽象机制 ⚡

---

## 实体类型

### RbacGlobalEntity（全局实体）

**解释**：跨租户共享的权限配置数据，所有租户使用同一份数据。

| 实体 | 说明 |
|------|------|
| `RbacApiGroup` | API 分组 |
| `RbacApiPermission` | API 权限定义 |
| `RbacApiGroupMapping` | API 与分组关联 |
| `RbacFeaturePermission` | 功能权限（树形结构） |
| `RbacFeatureApiMapping` | 功能与 API 关联 |
| `RbacDataDimension` | 数据权限维度定义 |
| `RbacDataDimensionMapping` | 维度与业务表映射 |

**维护者**：通常由系统管理员维护

---

### RbacEntity（租户实体）

**解释**：需要按租户隔离的业务数据，每个租户独立管理。

| 实体 | 说明 |
|------|------|
| `RbacUser` | 用户 |
| `RbacUserProfile` | 用户扩展信息 |
| `RbacRole` | 角色 |
| `RbacUserRole` | 用户角色关联 |
| `RbacRoleApiPermission` | 角色 API 权限关联 |
| `RbacRoleFeaturePermission` | 角色功能权限关联 |
| `RbacDataScope` | 数据范围值 |
| `RbacRoleDataScope` | 角色数据权限关联 |
| `RbacRoleDataScopeFlag` | 角色数据权限特殊标记 |

**隔离方式**：通过 TenantId 字段进行租户隔离

---

## 权限计算

### 并集（Union）

**解释**：多角色权限合并方式，角色越多，权限范围越大。

```
用户角色A权限：[接口1, 接口2]
用户角色B权限：[接口2, 接口3]

用户最终权限 = [接口1, 接口2, 接口3]（并集）
```

---

### 交集（Intersection）

**解释**：数据查询过滤方式，用户权限范围 ∩ 实际数据。

```
用户权限范围：研发部、市场部
实际数据：研发部、市场部、财务部

查询结果：研发部、市场部（交集）
```

---

### AND 关系

**解释**：多维度数据权限的组合方式，必须同时满足所有维度。

```
用户权限：
  - 部门维度：研发部
  - 区域维度：华东区

查询条件：department_id = 研发部 AND region_id = 华东区
```

---

## 数据权限特殊标记

| 标记 | 值 | 说明 | 适用场景 |
|------|-----|------|---------|
| 全部 | `ALL` | 可访问该维度下所有数据 | 部门负责人可查看全部数据 |
| 仅自己 | `SELF` | 仅可访问自己创建的数据 | 普通员工只能看自己创建的数据 |
| 自定义 | `CUSTOM` | 按具体勾选的维度值过滤 | 按需分配具体数据范围 |

**优先级**：ALL > SELF > CUSTOM

---

## 功能权限类型

### FeatureType（功能类型）

树形结构中的节点类型：

| 类型 | 英文 | 说明 |
|------|------|------|
| 模块 | `MODULE` | 顶级菜单，如"系统管理" |
| 页面 | `PAGE` | 二级菜单，如"用户管理" |
| 按钮 | `BUTTON` | 页面内操作按钮，如"新增用户" |

---

### PermissionType（权限类型）

API 权限的授权类型：

| 类型 | 说明 |
|------|------|
| `API` | 单个 API 接口 |
| `API_GROUP` | API 分组（批量授权） |

## 核心机制

### 显式调用

**解释**：数据权限需要开发者主动调用 `.WithDataScope()` 才生效，不自动附加。

```csharp
// ✅ 显式调用 - 明确知道这里控制了数据权限
var orders = await _dbContext.Orders
    .WithDataScope()
    .ToListAsync();

// ✅ 不调用 - 不过滤（特殊场景）
var allOrders = await _dbContext.Orders.ToListAsync();
```

**优点**：
- 代码可读性强，一眼看出哪里控制了数据权限
- 灵活自由，不调用就是不过滤
- 避免在复杂业务中"帮倒忙"

---

### 精确授权

**解释**：勾选哪个节点就授权哪个，不自动继承子级。

```
勾选"用户管理"页面 → 只授权 PAGE 级别
    → 不自动授权"新增用户"按钮
    → 需要显式勾选按钮
```

---

### 软删除

**解释**：删除操作仅标记 `IsDeleted = true`，不物理删除数据。

**实现**：通过 EF Core 全局查询过滤器自动过滤已删除记录。

---

### 超级管理员

**解释**：具有特殊权限的用户，跳过所有权限校验。

**标识**：`User.IsSuperAdmin = true`

**特性**：
- 自动拥有所有 API、功能、数据权限
- 数据权限过滤对其不生效

---

## 技术术语

### 租户隔离

**解释**：多租户系统中，每个租户的数据相互隔离，互不可见。

**实现方式**：通过 `TenantId` 字段进行数据隔离

---

### 全局查询过滤器

**解释**：EF Core 特性，自动为所有查询追加过滤条件。

**用途**：实现软删除、租户隔离等全局过滤逻辑。

---

### 表达式树

**解释**：.NET 中表示代码结构的数据结构，用于动态构建查询条件。

**用途**：数据权限过滤时动态生成 WHERE 条件。
